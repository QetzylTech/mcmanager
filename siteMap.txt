Suggested Site Map / Navigation Hierarchy

Purpose
- Organize the current prototype, legacy carry-over features, and the next-generation multi-world requirements into one navigation plan.
- Preserve the strengths of the old shell layout while making room for multi-world, multi-version, and future multi-node growth.

Navigation principles
- Keep the shell model from the legacy app:
  - shared nav
  - shared alerts
  - shared live state ownership
  - page-level content panes
- Separate manager-wide pages from per-world pages.
- Avoid putting everything on Home once multiple worlds exist.
- Make destructive or advanced actions live near the thing they affect.
- Keep read-only operational views easy to reach from anywhere.

- Dashboard
- Worlds

1. Dashboard
- Purpose:
  - Manager-wide summary
  - Fast operational awareness
  - Primary landing page
- Content:
  - Manager health summary
  - Host storage / CPU / RAM
  - Global alerts
  - Worlds overview cards
  - Recently active worlds
  - Recent backups / restores / failures
  - Quick actions
- Suggested panes:
  - Global Status
  - World Summary
  - Active Alerts
  - Recent Activity
  - Quick Actions
- Notes:
  - This replaces the old single-world Home as the true landing page.
  - The current prototype Home layout can still be reused visually, but its content should become manager-wide.

2. Worlds
- Purpose:
  - Main entry point for all managed Minecraft worlds
  - Central place to browse, create, and select worlds
- Children:
  - World List
  - Create World
  - Templates / Server Bases
  - Java Runtimes

2.1 World List
- Content:
  - Search
  - Filters
  - Status badges
  - Version badges
  - Server type badges
  - Host / node ownership
  - Address / connection target
  - Quick actions
- Actions:
  - Open world detail
  - Start
  - Stop
  - Backup
  - Open logs

2.2 Create World
- Content:
  - World name
  - World ID preview
  - Version selection
  - Server type / baseline selection
  - Java runtime selection
  - Port / address allocation
  - Storage path
  - Initial backup / retention defaults

2.3 Templates / Server Bases
- Content:
  - Curated Vanilla baselines
  - Forge baselines
  - Fabric baselines
  - Future custom baselines

2.4 Java Runtimes
- Content:
  - Registered Java runtimes
  - Version mapping
  - Default runtime
  - Per-world compatibility notes

Suggested phase plan
- Phase 1:
  - Keep current layout style
  - Rename Home -> Dashboard
  - Add Worlds page
  - Move current single-world Home into World Detail -> Overview
- Phase 2:
  - Split logs into manager-wide Logs page and per-world Logs tab
  - Split Cleanup into manager-wide Maintenance and per-world Maintenance tab
- Phase 3:
  - Add Activity & Alerts as a dedicated area
  - Expand Settings into multi-section configuration

Open sitemap questions
- Should Dashboard be purely manager-wide, or should it have a selected-world focus mode?
- Should World Detail use tabs, left-side subnav, or pane switcher buttons?
- Should Backups & Restore remain one page, or split into list and restore workflow pages?
- Should Activity & Alerts be top-level, or live under Dashboard with deeper drill-down pages?
- Should Setup remain top-level after onboarding, or become a section within Settings?
