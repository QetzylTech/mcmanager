Legacy System Study: `legacyCode`

Scope and baseline
- Studied the frozen legacy implementation under `legacyCode`.
- Compared `project requirements.txt` at repo root with `legacyCode/doc/project requirements.txt`; they are the same document in practice, so the checklist below compares one requirements baseline against the legacy implementation.
- Main entrypoints:
  - `legacyCode/mcweb.py`
  - `legacyCode/app/main.py`
  - `legacyCode/app/application_factory.py`
- Main architecture contract:
  - `legacyCode/ARCHITECTURE.md`


Directory structure and use

- `legacyCode/`
  - Frozen old application root.
  - Contains the Flask app, static frontend, templates, docs, scripts, tests, debug helpers, and nginx boot helpers.

- `legacyCode/app/`
  - Main Python application package.
  - Real implementation lives here, not in `mcweb.py`.

- `legacyCode/app/bootstrap/`
  - App composition and startup wiring.
  - Loads `mcweb.env`, evaluates setup mode, builds runtime state, bootstraps data, wires services, and decides whether to run `web`, `worker`, or `all`.
  - Key files:
    - `web_app.py`: builds the Flask app and runtime.
    - `web_app_config.py`: bootstrap config loading and Flask secret key setup.
    - `web_app_state.py`: creates the giant runtime state bundle.
    - `web_app_setup.py`: setup-mode route guard and setup submission wiring.
    - `worker_app.py`: worker-only entrypoint.

- `legacyCode/app/routes/`
  - HTTP translation layer.
  - Routes are intentionally thin and call commands or queries.
  - Key route modules:
    - `dashboard_routes.py`: shell, docs, diagnostics, nav-alert endpoints.
    - `dashboard_control_routes.py`: start/stop/backup/restore/RCON.
    - `dashboard_metrics_routes.py`: `/metrics` and `/metrics-stream`.
    - `dashboard_notifications_routes.py`: notifications SSE and operation SSE.
    - `dashboard_file_routes.py`: backups/log pages, downloads, file viewing, log SSE.
    - `dashboard_maintenance_api_routes.py`: cleanup page, cleanup SSE, cleanup APIs.
    - `panel_settings_routes.py`: admin panel settings and device-map management.
    - `setup_routes.py`: first-run setup page and validation endpoints.
    - `shell_page.py`: fragment-vs-shell rendering helper.

- `legacyCode/app/commands/`
  - Write-side application commands.
  - Handles validation, idempotency, operation records, queuing, and command-side conflict enforcement.
  - Key modules:
    - `control_handlers.py`: start/stop/backup/restore/RCON flows.
    - `control_support.py`: shared operation helpers, rate limiting, worker-thread launch.
    - `maintenance_commands.py`: cleanup password checks, rule save/run/delete/ack.
    - `setup_commands.py`: setup form validation and save orchestration.

- `legacyCode/app/queries/`
  - Read-side models and validation helpers.
  - Key modules:
    - `dashboard_queries.py`: shell alerts, observed-state, consistency report.
    - `dashboard_file_queries.py`: safe log/file lookup and content reads.
    - `maintenance_queries.py`: cached maintenance page payloads and async refresh.
    - `setup_queries.py`: setup validation and timezone options.

- `legacyCode/app/services/`
  - Business logic and background runtime behavior.
  - This is the largest layer.
  - Major responsibilities:
    - control-plane use cases (`start_usecase.py`, `stop_usecase.py`, `backup_usecase.py`)
    - restore flow (`restore_execution.py`, `restore_jobs.py`, `restore_status.py`, `restore_workflow_helpers.py`)
    - metrics/log/file runtimes (`dashboard_metrics_runtime.py`, `log_stream_service.py`, `dashboard_file_runtime.py`)
    - observed-state and operation reconciliation (`dashboard_state_runtime.py`, `dashboard_operations_runtime.py`)
    - maintenance system (`maintenance_*`)
    - setup and boot data preparation (`setup_service.py`, `data_bootstrap.py`)
    - watcher loops (`session_watchers.py`, `worker_runtime.py`, `worker_scheduler.py`)

- `legacyCode/app/core/`
  - Lower-level app utilities and persistence modules.
  - Includes:
    - `state_store_*`: SQLite schema and data access
    - `response_helpers.py`: standard Flask responses
    - `security.py`, `rate_limit.py`
    - `filesystem_utils.py`
    - `action_logging.py`
    - `web_config.py`: dotenv-style config parser
    - `profiling.py`

- `legacyCode/app/state/`
  - Typed runtime state containers.
  - `AppState` splits state into:
    - immutable config
    - mutable runtime values
    - callable service ports
    - compatibility `ctx`

- `legacyCode/app/ports/`
  - Port interfaces and default port registry.
  - Defines abstractions for:
    - service control
    - metrics
    - logs
    - backup execution
    - state store
    - filesystem

- `legacyCode/app/infrastructure/`
  - Concrete adapters implementing ports.
  - Bridges service logic to platform modules and the SQLite store.

- `legacyCode/app/platform/`
  - OS-specific implementations.
  - `loader.py` selects modules by OS.
  - Current legacy control path still depends on platform-specific service control, especially systemd on Linux.
  - Important implication:
    - the old system did not finish the future-plan goal of removing system service dependence.

- `legacyCode/templates/`
  - Flask templates.
  - `app_shell.html`: persistent shell wrapper.
  - `setup.html`: setup page.
  - `fragments/*.html`: page fragments mounted inside the shell.
  - `partials/nav.html`, `partials/theme_init.html`: shared shell pieces.

- `legacyCode/static/`
  - Frontend runtime and page scripts.
  - Important files:
    - `app_shell.js`: shell navigation, SSE ownership, multi-tab coordination, view-state persistence.
    - `service_worker.js`: offline shell and fragment fallback.
    - `offline_recovery.js`: offline banner and reconnect flow.
    - `dashboard_home_page.js`, `file_browser_page.js`, `maintenance_page.js`, `panel_settings.js`, `documentation_core.js`: page-level behavior.

- `legacyCode/tests/`
  - Large regression suite.
  - Covers routes, architecture boundaries, performance guards, control semantics, restore, maintenance, setup, metrics, and runtime behavior.

- `legacyCode/doc/`
  - Requirements, quality gates, samples, setup docs.

- `legacyCode/scripts/`
  - External helpers.
  - `backup.sh` is a major dependency for real backup execution.
  - also contains performance scripts and topology export/import helpers.

- `legacyCode/debug/`
  - Separate debug mini-app and debug-only tooling.
  - Debug routes are hidden behind the `DEBUG` config path described in the legacy README.

- `legacyCode/nginx/`
  - Optional reverse-proxy / boot-animation / wake helper experiments.
  - Not part of the main Flask app runtime.


Old software architecture

High-level layer model
- Intended dependency direction from `ARCHITECTURE.md`:
  - `routes -> commands/queries -> services -> ports -> infrastructure -> platform`
- Actual legacy code mostly follows that.
- The app uses one large shared runtime state object, but wraps it in typed contexts:
  - `ConfigContext`
  - `RuntimeContext`
  - `ServicePorts`
  - `UnifiedServiceContext`

Boot and composition flow
- `mcweb.py` or `python -m app.main`
  - imports `create_app()` or `run_server()`
- `app/bootstrap/web_app.py`
  - loads typed config from `mcweb.env`
  - creates the Flask app
  - builds runtime state values in `web_app_state.build_state()`
  - ensures bootstrap data files and SQLite schema exist via `data_bootstrap.ensure_data_bootstrap()`
  - evaluates whether setup mode is required
  - builds runtime bindings with `runtime_wiring.create_runtime()`
  - registers routes
  - exposes:
    - `run_server()`
    - `run_worker()`
    - `ensure_runtime_bootstrapped()`

Runtime roles
- `MCWEB_PROCESS_ROLE=all|web|worker`
  - `all`: web process plus background loops in one process
  - `web`: request-serving only, no business background ownership beyond request/route work
  - `worker`: background loops only; consumes queued intents and precomputes/publishes state

Request architecture
- Browser shell requests a page.
- Route returns either:
  - full `app_shell.html`, or
  - fragment-only HTML when `X-MCWEB-Fragment: 1`
- Page JS mounts inside the persistent shell.
- Writes go through command handlers.
- Reads go through query helpers or cached runtime functions.

Control-plane architecture
- Protected/async actions are operation-based.
- Operations are recorded in SQLite `operations`.
- Control intents are appended to SQLite `events` topic `control_intent`.
- In `all` mode, request handlers often start worker threads immediately.
- In `web` + `worker` split mode:
  - web process records the intent
  - worker process consumes it from SQLite
  - worker updates the operation status back into SQLite
- Operation progress is exposed through:
  - `/operation-status/<op_id>`
  - `/operation-stream`
  - restore-specific status/log SSE

Streaming architecture
- Server uses SQLite-backed event topics plus in-memory caches.
- Main topics:
  - `metrics_snapshot`
  - `ui_notification`
  - `operation_update`
  - `restore_log`
  - `log:<source>`
  - `maintenance_state:<scope>`
  - `control_intent`
- SSE endpoints read these topics and/or in-memory conditions.

Frontend architecture
- Persistent shell:
  - `app_shell.html`
  - `app_shell.js`
- Shell owns:
  - navigation
  - theme
  - metrics SSE
  - notifications SSE
  - operation SSE
  - multi-tab primary election
  - shared home log streams
  - cached page data
  - offline recovery hooks
- Page modules own only mount/unmount and page-specific DOM logic.

Multi-tab architecture
- One tab becomes the primary tab using:
  - `localStorage` primary record
  - `BroadcastChannel` when available
  - `storage` event fallback when BroadcastChannel is unavailable
- Only the primary tab keeps the live shared SSE streams open.
- Secondary tabs receive forwarded snapshots/events from the primary tab.

Persistence architecture
- Config:
  - `mcweb.env`
- Structured state:
  - `data/app_state.sqlite3`
- Small runtime text/json files:
  - `data/session.txt`
  - `data/state.txt`
  - `data/cleanup_non_normal.txt`
  - `data/restore.history`
- Log files:
  - `logs/backup.log`
  - `logs/restore.log`
  - `logs/mcweb.log`
  - `logs/mcweb_actions.log`
- Managed world/backup data:
  - minecraft root directory
  - backup directory
  - snapshot directory
  - `data/old_worlds`

OS boundary
- Services do not directly call `systemctl`, `journalctl`, `mcrcon`, disk usage, or shell utilities.
- They call ports.
- Ports are implemented by infrastructure adapters.
- Adapters call platform modules chosen by `app/platform/loader.py`.
- Important current reality:
  - Linux service control still uses `systemctl`.
  - Linux live log streaming still uses `journalctl`.

End-to-end runtime flow
- Browser load
  - Client requests a route such as `/` or `/backups`.
  - Flask returns the full shell for first load, then fragment HTML for internal navigation.
  - `app_shell.js` mounts the fragment into the persistent shell and restores local UI state.
- Live data ownership
  - The primary tab opens the shared SSE streams.
  - Secondary tabs receive rebroadcasted payloads through `BroadcastChannel` or `storage` events.
  - The shell updates shared UI state, notifications, nav alerts, and cached snapshots.
- Action request flow
  - A user action such as start, stop, backup, restore, or cleanup hits a thin route.
  - The route calls a command handler that validates passwords, storage guards, conflicts, cooldowns, and idempotency.
  - The command writes durable operation state to SQLite and may also publish an event intent.
- Background execution
  - In single-process `all` mode, the app can execute work immediately in background threads.
  - In split `web` + `worker` mode, the web process records intents and the worker process consumes them.
  - Worker/services call ports, which resolve to infrastructure adapters and then platform-specific implementations.
- Feedback loop
  - Services update operation rows, publish event topics, append logs, and refresh caches.
  - SSE endpoints stream those updates back to the shell.
  - The shell rehydrates the visible page and forwards the same data to secondary tabs.

Background worker map
- `metrics_collector`
  - publishes dashboard metrics snapshots
- `operation_reconciler`
  - closes stale or observed operations
- `file-page-cache-refresher`
  - refreshes backups/log file lists
- `idle_player_watcher`
  - 0-player auto-stop
- `backup_session_watcher`
  - periodic backup and shutdown backup logic
- `storage_safety_watcher`
  - emergency shutdown when storage is unsafe
- `cleanup_scheduler`
  - scheduled cleanup triggers
- `log_stream_fetcher_<source>`
  - one fetcher per log source
- worker-only loops from `worker_runtime.py`
  - control intent consumer
  - maintenance precompute publisher
  - inventory index refresh


Runtime data model and state files

- `mcweb.env`
  - primary runtime config source
  - stores service name, password hashes, paths, timing knobs, timezone, secret key, role

- `data/session.txt`
  - session start timestamp for the current Minecraft session
  - used for session duration and periodic backup scheduling

- `data/state.txt`
  - backup-running flag written by the external backup system
  - app also treats the in-memory backup run lock as an additional single-flight guard

- `data/cleanup_non_normal.txt`
  - missed cleanup run tracking and acknowledgements

- `data/restore.history`
  - append-only restore history text log

- `logs/*.log`
  - persistent app-side logs surfaced in the UI and through SSE/file viewers


Database schema (`data/app_state.sqlite3`)

Schema characteristics
- Created by `app/core/state_store_core.py`.
- Uses SQLite.
- Enables WAL when possible, falls back to DELETE journal when the filesystem rejects WAL.
- No explicit foreign keys.
- Consistency is mostly enforced in application code.

- There are backward-compatible migrations for:

Implementation map

1. Bootstrap and composition

- Module: `app/bootstrap/web_app.py`
  - Feature: app startup composition
    - Functions:
      - `load_bootstrap_config`
      - `build_state`
      - `ensure_data_bootstrap`
      - `configure_setup`
      - `create_runtime`
      - `run_server`
      - `run_worker`
    - Steps:
      - load config from `mcweb.env`
      - create Flask app
      - build state values, locks, caches, file paths, and timing knobs
      - ensure `data/` files and SQLite schema exist
      - decide whether setup mode is required
      - wire runtime bindings and routes
      - start boot tasks and server, or start worker-only loops
    - Conditions:
      - if setup is required, runtime boot is reduced and `/setup` is forced
      - if process role is `worker`, Flask routes are not the main purpose
    - Edge cases:
      - incompatible data files are archived to `data/old_app_data`
      - active restore operations are force-failed on reboot bootstrap

- Module: `app/bootstrap/web_app_state.py`
  - Feature: central runtime state creation
    - Functions:
      - `build_state`
    - Steps:
      - derive paths for logs/data/docs/runtime files
      - create locks and caches for backup, session, logs, metrics, restore, storage, device map, password throttle, clients
      - define service status intent and log-stream states
    - Conditions:
      - world path is refreshed later from `server.properties`
    - Edge cases:
      - defaults may not reflect the true active world until `world_bindings` refreshes them


2. Setup mode and configuration

- Page/Module: Setup page (`/setup`)
  - Feature: detect missing or invalid runtime config
    - Functions:
      - `setup_service.assess_setup_requirement`
      - `configure_setup`
    - Steps:
      - if `mcweb.env` is missing or required values are invalid, setup mode becomes active
      - non-setup requests are redirected to `/setup`
    - Conditions:
      - `paths_only` mode is used when only the path values are invalid/missing
      - `full` mode is used when password/timezone/core config is missing or invalid
    - Edge cases:
      - invalid timezone or OS-invalid paths force setup

  - Feature: validate setup input without saving
    - Functions:
      - `setup_queries.validate_setup_request`
      - route `/setup/validate`
    - Steps:
      - validate timezone
      - validate Minecraft root path
      - validate backup path and optional creation allowance
    - Conditions:
      - different validation kind per request: `timezone`, `root`, `backup`
    - Edge cases:
      - backup path can report missing-but-creatable separately

  - Feature: save setup values
    - Functions:
      - `setup_commands.handle_setup_submit`
      - `setup_orchestration.save_setup_values`
      - `setup_service.write_env_file`
    - Steps:
      - normalize form payload
      - validate required fields
      - validate password length/match when in full mode
      - generate password hash
      - write `mcweb.env`
      - ensure bootstrap data exists
      - trigger process reload
    - Conditions:
      - full mode requires password and confirm
      - paths-only mode reuses existing hashes
    - Edge cases:
      - data residuals can be archived when setup rewrites the app state


3. Shell and navigation architecture

- Page/Module: Shared shell (`app_shell.html`, `static/app_shell.js`)
  - Feature: persistent shell with fragment navigation
    - Functions:
      - `shell_page.fragment_response_requested`
      - `shell_page.render_shell_page`
      - `app_shell.js` navigation/mount logic
    - Steps:
      - full request returns `app_shell.html` with initial fragment HTML
      - internal navigation sends `X-MCWEB-Fragment: 1`
      - response fragment replaces `#mcweb-app-content`
      - page module unmounts previous page and mounts next page
    - Conditions:
      - if fragment header is present, Flask returns fragment only
    - Edge cases:
      - navigation failure triggers offline recovery helper

  - Feature: multi-tab single-owner live runtime
    - Functions:
      - `BroadcastChannel` logic in `app_shell.js`
      - primary-tab election helpers
    - Steps:
      - tabs elect a primary owner
      - primary tab owns metrics/notifications/operation/home-log streams
      - primary forwards snapshots and events to secondary tabs
    - Conditions:
      - uses `BroadcastChannel` if supported
      - falls back to `localStorage` storage events otherwise
    - Edge cases:
      - stale primary ownership expires using TTL heartbeat logic

  - Feature: offline shell and recovery
    - Functions:
      - `service_worker.js`
      - `offline_recovery.js`
      - route `/sw.js`
    - Steps:
      - precache shell assets and common routes/fragments
      - serve cached HTML or fragment when server is offline
      - show offline banner and attempt recovery/reload on reconnect
    - Conditions:
      - fragment requests get a cached fragment/offline fragment
      - navigations get cached page/offline page fallback
    - Edge cases:
      - if cache is missing, service worker returns a plain 503 response


4. Home page and shared runtime state

- Page/Module: Home page (`/`)
  - Feature: render shell with initial metrics and initial log snapshots
    - Functions:
      - `dashboard_routes.index`
      - `dashboard_queries.get_dashboard_shell_model`
    - Steps:
      - build alert message from query string message code
      - gather cached log snapshots for `minecraft`, `backup`, `mcweb`, `mcweb_log`
      - render home fragment inside shell
    - Conditions:
      - shell request vs fragment request changes the response wrapper
    - Edge cases:
      - if cached logs fail to load, empty strings are used

  - Feature: home-page activity heartbeat
    - Functions:
      - route `/home-heartbeat`
      - `dashboard_metrics_runtime.mark_home_page_client_active`
    - Steps:
      - touch client activity
      - update `home_page_last_seen`
      - wake metrics collector waiters
    - Conditions:
      - optional `client_id`
    - Edge cases:
      - marker handles both legacy no-arg and new client-id paths

  - Feature: diagnostics and shell support endpoints
    - Routes/functions:
      - `/observed-state`
      - `/consistency-check`
      - `/profiling-summary`
      - `/device-name-map`
      - `/ui-error-log`
    - Steps:
      - expose runtime-observed state
      - expose safe auto-repair consistency report
      - expose profiling only when enabled and password-approved
      - expose IP/device mapping for UI
      - log client modal/UI failures to server logs
    - Conditions:
      - profiling and auto-repair require password
    - Edge cases:
      - observed state is cached briefly, so UI can lag by about 1 second


5. Metrics and dashboard live state

- Module: `app/routes/dashboard_metrics_routes.py`
  - Feature: one-shot metrics snapshot
    - Route/function:
      - `/metrics`
    - Steps:
      - best-effort refresh live metrics
      - try latest SQLite `metrics_snapshot`
      - fall back to in-memory cached snapshot
    - Conditions:
      - short per-route cache avoids burst overload
    - Edge cases:
      - that route cache can slightly delay visible transitions

  - Feature: metrics SSE stream
    - Route/function:
      - `/metrics-stream`
    - Steps:
      - register client
      - increment metrics stream count
      - send initial snapshot from memory or SQLite
      - wait on condition variable and also poll SQLite event topic
      - send keepalives when nothing changed
    - Conditions:
      - nav-attention payload may be merged into the snapshot
    - Edge cases:
      - stream closes unregister the client and decrement active stream count

- Module: `app/services/dashboard_metrics_runtime.py`
  - Feature: metrics collection cadence
    - Functions:
      - `has_active_flask_app_clients`
      - `metrics_collector_loop`
      - `collect_and_publish_metrics`
    - Steps:
      - if no active clients, collector mostly sleeps
      - if server is on but no clients, storage/cache refresh can still happen at slow cadence
      - if active clients exist, collect and publish at normal cadence
      - publish into SQLite topic `metrics_snapshot` and in-memory condition state
    - Conditions:
      - worker role can collect continuously
      - active/off/offline intervals differ
    - Edge cases:
      - idle storage cache is refreshed separately so reconnecting clients see newer disk/cleanup counts

  - Feature: metric composition
    - Functions:
      - `collect_dashboard_metrics`
      - `get_slow_metrics`
      - `_resolve_service_status_display`
    - Steps:
      - collect CPU/RAM/frequency/storage via metrics port
      - collect RCON player/tick info
      - compute backup schedule, backup status, stale world count, cleanup metadata
      - derive UI classes and nested group payloads
    - Conditions:
      - active clients force fresher slow-metric refresh
    - Edge cases:

Bottom line
- The legacy system is not a small Flask app anymore; it is already a layered control-plane application with:
  - a persistent SPA-like shell
  - SQLite-backed operation/event streams
  - split web/worker runtime support
  - background watchers and schedulers
  - setup, panel settings, restore orchestration, cleanup engine, and multi-tab/offline behavior
- The biggest architectural constraints carried forward from the old system are:
  - one very large shared runtime state object
  - reliance on external scripts and platform service commands
  - partial transactional safety across filesystem-heavy operations
  - a lot of capability concentrated into one application process model, even though the code has started splitting concerns into layers


Known gaps / uncertain areas

- This document is a high-confidence architecture and behavior map, not a file-by-file or symbol-by-symbol index of every source file in `legacyCode`.
- Large modules, especially in `app/services/` and `static/`, were analyzed by behavior and major control flow rather than by exhaustively documenting every helper function.
- Frontend visual compliance against the full UI spec was only partially verified; exact CSS tokens, spacing rules, modal variants, and some interaction details may differ from the requirements text in ways not fully captured here.
- The test suite was not fully mapped into this document, so the file does not yet show feature-to-test coverage or identify which requirements are protected by automated tests versus only implemented in code.
- Debug-only code, optional nginx helpers, and some support scripts were reviewed at a high level but not documented with the same depth as the main Flask runtime.
- Some behavioral conclusions are inferred from the interaction of routes, services, caches, and worker loops rather than from a single explicit specification inside the legacy code.
- Certain UX requirements, especially popup timing, form-reset behavior, exact button enable/disable states, and some alert visuals, were not exhaustively verified across every page script and template.
- Restore safety was reviewed carefully, but filesystem-heavy operations can contain edge cases that only appear with real corrupted archives, unusual path layouts, partial disk failures, or race conditions that were not simulated in this pass.
- Platform behavior outside the main Linux path may be less certain because the legacy implementation is clearly optimized around Linux service control and journal access.
- If you want this file to become near-exhaustive, the next upgrade would be a file-by-file inventory, route-to-template/script mapping, module-level function inventories, and a test coverage map.
